# Extract Shared Cache Key Normalizer

## Context

The cache key normalization logic is duplicated across multiple API routes:

```typescript
// In /api/lyrics/route.ts AND /api/concerts/route.ts (identical code):
.normalize('NFKD')
.replace(/[\u0300-\u036f]/g, '')
.replace(/[^a-zA-Z0-9\s]/g, ' ')
.replace(/\s+/g, ' ')
.trim()
.toLowerCase()
```

This function strips diacritics, removes special characters, and normalizes whitespace to create consistent cache keys for artist names and song titles. Having it duplicated means a fix in one place won't propagate to the other, and new routes that need cache keys will likely copy-paste again.

## Directive

1. **Create `src/lib/cache-keys.ts`** with:
   ```typescript
   export function normalizeCacheKey(input: string): string
   ```
   Move the shared normalization logic here.

2. **Add specialized key builders**:
   ```typescript
   export function lyricsKey(artist: string, title: string): string
   export function concertsKey(artist: string): string
   export function artistInfoKey(artist: string): string
   export function itunesKey(term: string, media: string): string
   ```
   Each uses `normalizeCacheKey()` internally with the appropriate key format.

3. **Update all routes** to import from the shared module:
   - `/api/lyrics/route.ts` — use `lyricsKey()`
   - `/api/concerts/route.ts` — use `concertsKey()`
   - `/api/artist-info/route.ts` — use `artistInfoKey()`
   - `/api/itunes/route.ts` — use `itunesKey()`

4. **Update cron sync route** if it has its own key normalization.

**Boundaries:**
- Do NOT change the normalization algorithm — keep exact same behavior.
- Do NOT modify cache TTLs or cache logic.
- Only extract and centralize the key generation.

## Acceptance Criteria

- [ ] `src/lib/cache-keys.ts` exists with `normalizeCacheKey()` and 4 specialized builders.

- [ ] Cache keys generated by the new module match the old ones exactly (same normalization).
- [ ] All 4 API routes + cron sync import from the shared module.
